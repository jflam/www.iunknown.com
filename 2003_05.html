Hello Movable Type!

After more pain than I thought that I would go through, I finally finished converting my old CityDesk site over to Movable Type. The pain was largely due to two fairly major problems that I encountered along the way.

The first problem was due to the way that Movable Type wanted to maintain my entry archives. I used the individual entry archive option, which results in the generation of archive entries like "000001.html". However, I wanted to make sure that I didn't break any external links to my existing archive entries which had names like "http://www.iunknown.com/Archives/fog0000000030.html". To map between the two, I used an Apache .htaccess file to map the old URL's to the new URL's.

While I only have 257 entries in my weblog, I really didn't want to create my .htaccess file by hand. To solve this problem, I needed to write some code that would let me export the existing URL's for my archive entries from CityDesk. I wrote earlier about Brian Cantoni's excellent script for exporting entries. I had to modify Brian's script somewhat (for example, I export my entries by ascending date/time since I wanted my weblog entry to be "000001.html" and not "000257.html"). Also, I needed to generate XML in my export script since I would run over that script using a C# app that I wrote to solve nasty problem #2. Here's the CityDesk script that generates an export file called entries.xml:



<?xml version="1.0" encoding="windows-1252"?>

{$ setDateTimeFormat "English" "MM/dd/yyyy" "hh:mm:ss tt"$}



{$foreach x in (folder "Weblog") SortAscendBy .filedDate$} 

<entry>

  author>John</author>

  <title><![CDATA[{$x.headline$}]]></title>

  <date>{$x.fileddate$} {$x.filedtime$}</date>

  <status>Publish</status>

  <link>{$x.link$}</link>

  <body><![CDATA[{$x.body$}]]></body>

</entry>

{$ next $}

</entries>

The second problem had to do with my comments. For my old site, all of my comments were managed by Haloscan, who for over a year provided me with comments on my web site. Unfortunately, they don't currently offer an export option to dump all of my existing comments out as an RSS file; I had to write some code to screen-scrape their existing site. Fortunately for me, most of their comment pages are valid XHTML, so I was able to do the requisite screen-scraping using XPath. Unfortunately, they don't correctly escape "&" and a few other characters so some of my comment pages didn't migrate automatically over. I still need to manually migrate those comments over to MT.

If you're interested in the code that I wrote to manage my migration, you can download it here.

One final note: I posted this entry using w.bloggar, which is quite a nice offline editor. One thing that would really improve w.bloggar would be to use Microsoft's DHTML editing component as the editor within w.bloggar, much like what CityDesk does already.

Upgrading Windows 2000 Domain to Windows Server 2003

A bit of preamble: my existing Windows 2000 domain controller's hard drive is about to die (thanks to the SMART diagnostics built into the drive). It also houses my Exchange server, so I needed to rebuild an Exchange server on another box and migrate my email over to that box. The other box happens to be a Windows 2003 Server box that I had built some time ago, and it is currently just loafing around as my Perforce source control server. So, while I was at it I decided to update my domain to be a Windows 2003 domain.


This was, ahem, a somewhat painful process. Here's a summary of the bold strokes for anyone else who might be going down this path:



  • Run adprep /forestprep on your existing domain controller box

  • Run adprep /domainprep on your existing domain controller box

  • Run dcpromo on the Windows 2003 server box that you want to turn into an active directory server

All went fine in this process until I ran dcpromo. I then proceeded to get a very obscure (and useless) error message, which went along the lines of: "... unable to convert the computer account ZZZ$ to a domain controller account. Access is denied".


Now that's a pretty useless error message. After spending some quality time with Google, I found only one question from an unfortuate Russian poster on a newsgroup. There were, of course, no replies.


I then spent several hours trying various permutations until I hit upon the solution.



  • I thought that it might be a group membership problem, but I was trying to do the operation using a known Enterprise Admin account which has all of the appropriate other group memberships. Still, I double checked them.

  • I eventually realized that it might be a privilege problem, so I ran the Security Policy editor. The only privilege that my administrator account didn't have enabled was the "Enable computer and user accounts to be trusted for delegation" privilege. So I enabled this privilege.

  • When I re-ran dcpromo, it STILL didn't work, even though I had the privilege enabled.

  • I then went back to the Computers and Users management console and manually enabled "Trust computer for delegation" for my Windows 2003 Server computer. Once I did this and reran dcpromo, everything worked just fine.

I hope this helps someone else who runs into this problem. I sooo hate administration.

I might just run a sed/perl script over the XSD and convert all colons to underscores so that I can continue my work. But clearly, I would much prefer a world where GSLgen understood namespaces (and XPath too!).

Challenge-response email authentication

I've been thinking about a challenge-response email authentication scheme for quite a while. The idea is simple: the first time that someone who isn't on your whitelist sends you an email, they receive an automated response that says "click on this link to actually send me the email". If you don't blacklist that sender, all subsequent emails get through.


Well, apparently a company called MailBlocks has already done this. There are, however, two problems with their approach: 1) they don't integrate in with an existing enterprise-class email system and 2) they claim to hold the patent on the concept for which there is apparently ample evidence of prior art.


On a related note, my friend Aaron Skonnard has a similar setup on his voice line. If your number isn't on a recognized list of allowed callers or you are calling with caller id blocked, you will hear a recorded message that says "hit 1 if you are not a telemarketer". This has apparently blocked all unsolicited phone calls from his home, since it is illegal for a telemarketer to misrepresent themselves this way.

Generation Gap: Another approach to delegation in code generation

[Via Joel on Software] Folks that have been discussing code generation over on the Joel on Software forums pointed to a design pattern called Generation Gap, that was written up by John Vlissides that never made it into Design Patterns.


I don't use subclassing to do delegation; instead I call a static method on another class. For example:


public void MyGeneratedMethod( int arg1, float arg2, Stream arg3 )
{
  MyDelegatedClass.MyDelegatedMethod( this, arg1, arg2, arg3 );
}


I believe that my approach is more flexible in that I can delegate at any point in a class hierarchy, not just at the leaf nodes. The downside is that this can sometimes break encapsulation in systems that don't support the notion of internal visibility.

GSLgen, context and loops

In the first article in this series, I talked about the basics of acquiring, compiling and running GSLgen. In this article, I'll introduce GSLgen's notion of context, and how it can be used within loops.


Let's consider how we can use GSLgen to generate some C# classes. Here's an XML file that contains the simplest possible definition of two classes:


<classes>
  <class name="Person"/>
  <class name="Place"/>
</classes>


Here's a GSL script that iterates over the XML file:


using System;
.for classes.class


public class $(name)
{
}
.endfor


And this is what it generates:


using System;

public class Person
{
}


public class Place
{
}


Let's take a closer look at the GSL script. It showcases the for ... endfor construct which is commonly used to iterate over a set of XML elements. The for statement's expression, classes.class, defines what XML elements to iterate over. Within the body of the for ... endfor statement is a template. That template's context changes each time through the loop to point to the current XML element. The $(name) expression retrieves the value of the name attribute of the current XML node and outputs the value to the output stream.


Next, let's consider a more complex XML file that contains some nested elements that we want to iterate over:


<classes>
  <class name="Person">
    <field name="Name" type="string"/>
    <field name="Age" type="int"/>
  </class>
  <class name="Place">
    <field name="Address" type="string"/>
    <field name="Zip" type="string"/>
  </class>
</classes>


We can nest our for ... endfor loops like so:


using System;
.for classes.class


public class $(name)
{
.for field
  private $(type) $(name)_;
.endfor
}
.endfor


Which results in the generation of the following C# code:


using System;


public class Person
{
  private string Name_;
  private int Age_;
}


public class Place
{
  private string Address_;
  private string Zip_;
}


As you can see, the nested for ... endfor loop inherits the context of its parent for ... endfor loop. This lets us write very simple code that can iterate over nested elements using a very natural syntax.
