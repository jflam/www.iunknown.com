September 23, 2002

Flash

I went and looked in a section of the bookstore that I don't normally look at: the section on web design. You know the one, it was full of HTML books just 6 months ago. When I looked around today, I only saw a handful of HTML books. The rest, probaby 100+ titles or so, were ALL Flash books.


Could it be that Flash is now the dominant web development platform? After looking around at a few Flash community sites, I'm getting the distinct impression that Flash is rapidly replacing DHTML.


Macromedia claims better than 98% market penetration with Flash, which is largely due to very clever licensing deals with Microsoft. Since Flash is bundled with every copy of IE, as IE's market share grows, so does Flash's market share. However, with the amount of mind share that Flash has, it's quite remarkable that Macromedia as a company is doing so poorly, posting a loss of net loss of 3 cents a share for 1Q2003 on revenues of $83.7M.


It might have something to do with the ridiculous pricing of their toolkit. Flash MX sells for $499 US, and Macromedia Studio MX sells for $799 US. This has led to the rapid adoption of low-end tools like Swish, which boasts over 100,000 registered users of the $50 product.


Macromedia should wake up and smell the coffee: a $99 "standard" version of their Flash toolkit would make their toolkit so much more accessible to folks who are curious, and can only help to solidify their position as DHTML's main contender. Adobe gets it, with their $99 Photoshop Elements software. So does Microsoft, with their $99 "standard" versions of VB.NET, C# and VC++.

September 19, 2002

Intentional Programming

The recent announcement of Charles Simonyi founding Intentional Software with Gregor Kiczales is not that big of a surprise. Here's the facts: Microsoft killed the IP project at MSR, the two founders are both leaders in their respective (related) fields, and one of them is also capable of self-funding virtually any kind of start-up.


I've had the privilege of meeting some folks in the IP community, as well as the AOP community. I was very caught up in the AOP movement recently, and was quite convinced that it had a strong chance of being "the next big thing" in software development. I also had a chance to view Microsoft's famous "IP video", which is an amazingly well-produced documentary of what IP represented circa 2001.


Once I had a chance to calm down and take a step back from AOP to see the forest for the trees, I had a change of heart. This stuff is just too complicated. For most folks, the challenge will be dealing with cross-aspect interactions and / or creating aspects that are free of side-effects (and as a result being that much less interesting / useful). There are some AOP-like concepts in IP, which is why I suspect that Gregor was brought on board at IP.


I predict that AOP and IP (which has one of the coolest editors / IDE's that I have ever seen) will most likely be a niche market. I hope that the IP editor gets a life at Intentional Software, and that the world gets a chance to see this thing.


Does this change of heart mean that I'm abandoning my AOP work? No, but it means that I'm no longer trying to create a general-purpose AOP tool. The weaving engine continues to live on, but in a vastly different product space than what I had envisioned when I started my CLAW project. More on what that is when we get closer to a beta cycle.

Update on VB.NET code generation

Thanks to Scott Ellis who wrote in to explain why VB.NET is generating the extra local variable in a function declaration. It's due to the fact that you can assign the return value of the function to the name of the function in VB.NET. Scott sent along the following code snippet that illustrates the usage of the extra local:


Public Function AddThreeAndSevenAsn() As Integer
    Dim a As Integer = 3
    Dim b As Integer = 7
    AddThreeAndSevenAsn = a + b
End Function


When you disassemble this function using ILDASM, you'll get the following:


.method public instance int32  AddThreeAndSevenAsn() cil managed
{
  // Code size       11 (0xb)
  .maxstack  2
  .locals init ([0] int32 a,
           [1] int32 AddThreeAndSevenAsn,
           [2] int32 b)
  IL_0000:  nop
  IL_0001:  ldc.i4.3
  IL_0002:  stloc.0
  IL_0003:  ldc.i4.7
  IL_0004:  stloc.2
  IL_0005:  ldloc.0
  IL_0006:  ldloc.2
  IL_0007:  add.ovf
  IL_0008:  stloc.1
  IL_0009:  ldloc.1
  IL_000a:  ret
} // end of method Class1::AddThreeAndSevenAsn


This totally makes sense to me now.

Information Overload

My name is John, and I am an information junkie.


I came to this realization during my recent honeymoon trip. During this trip, I had zero connectivity to the outside world, an experience that I found to be unusually refreshing.


To try and preserve this feeling, I systematically unsubscribed from all of my mailing lists when I got home. Almost at once a sense of calmness settled over Outlook. After an initial adjustment period, I find that I no longer miss the daily torrent of messages that invade my mailbox. I am now free to spend my time focusing more intently at the tasks in front of me, instead of constantly being distracted by the dinging of the email bell.


With the installation of Spam Assassin Pro, I'm now also free from the 20 or so annoying pieces of spam that show up in my mailbox each day. I foolishly forgot to leave Outlook running on my PC at home, so as I write this from sunny(!) Kirkland, WA, I still have to delete my spam manually. You live and you learn.


Hopefully this post will help inspire others who have similarly suffered in silence. There is hope at the end of your SMTP channel, it's called UNSUBSCRIBE.


Peace.

September 06, 2002

Code generation: C# vs. VB.NET

I spent a bit of time today investigating the CIL generated by the VB.NET compiler vs. the CIL generated by the C# compiler. Consider the following VB.NET code:


  Public Function AddThreeAndSeven() As Integer
    Dim a As Integer = 3
    Dim b As Integer = 7
    Return a + b
  End Function


When run through vbc.exe using release settings, this is the CIL that is generated:


.maxstack 2
.locals (int V_0, int V_1, int V_2)
L_0000: ldc.i4.3
L_0001: stloc.0
L_0002: ldc.i4.7
L_0003: stloc.2
L_0004: ldloc.0
L_0005: ldloc.2
L_0006: add.ovf
L_0007: ret


Consider exactly the same function written using C#:


public int AddThreeAndSeven()
{
  int a = 3;
  int b = 7;
  return a + b;
}


This is the CIL that is generated:


.maxstack 2
.locals (int V_0, int V_1)
L_0000: ldc.i4.3
L_0001: stloc.0
L_0002: ldc.i4.7
L_0003: stloc.1
L_0004: ldloc.0
L_0005: ldloc.1
L_0006: add
L_0007: ret


As you can see, the VB.NET code allocates space for an additional local variable that is never used. This is behavior is consistent between debug and release builds. I'm rather surprised by this behavior, but I'm certain that the JIT catches the unused variable and allocates space in the stack frame appropriately.
