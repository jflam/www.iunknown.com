Let's be careful out there ...

I'm in the process of setting up a new desktop dev machine, and spent the better part of this evening installing tons of software. One of the patches that I hadn't installed yet was SQL Server 2000 Service Pack 3. The reason why I hadn't really thought about it was because none of my SQL Server boxes are exposed to the public Internet. My ISA server is configured to block UDP port 1434 traffic by default.


But there is one SQL Server box that I almost forgot about: my laptop. I tend to connect to all sorts of foreign networks with my laptop. Had I done so with an unpatched SQL Server running on my laptop, I would have brought the Slammer worm back home behind my ISA server firewall. Then all hell would have broken loose.


So the lesson here is: beware of laptops. They can easily subvert the security provided by your network's perimeter defenses. And make sure you patch ALL SQL servers regardless of whether they are behind your firewall or not!

January 24, 2003

Digital camera workflow

My brother Thomas and I bought a Canon S30 digital camera for our dad's birthday last week. I chipped in by donating my Epson Stylus 860 photo printer.


One of the things that new digital camera users inevitably run straight into is the complexity of the digital camera "workflow". The complexity of this workflow is not usually obvious until you compare it with the film camera "workflow":



  1. Shoot a roll of film
  2. Give it to someone else who processes your film into prints / slides

The digital camera workflow is significantly more complicated because you now are managing all of step 2 yourself. In a digital camera, your workflow looks like:



  1. Shoot some pictures
  2. Download pictures into your computer
  3. Select the ones that "turned out" and discard the rest
  4. Tweak and adjust the pictures
  5. Print the ones you really like

This is a lot more friction than the film workflow. Ideally, you would like to give your photos to your computer and have it perform the job of a digital mini-lab. While this is theoretically possible, it really isn't possible given the crappy usability of the software that exists today.


Steps 2 and 3 are handled reasonably well by software today. The things that complicate matters are steps 4 and 5. The problem here is that typical film camera users never could do these tasks in the past, so there is nothing for them to relate to.


Ideally, Step 4 would be fully automatic. It would detect common exposure flaws (red-eye caused by on-camera flashes, and shooting back-lit portraits without flash are probably the most common) and perform automatic exposure compensation. Eliminating Step 4 would be a huge boom to usability, as it eliminates a common "friction" step.


Step 5 can also be problematic, especially if you haven't color-calibrated your monitor and printer. If you want to "adjust" the colors of your photographs, invariably you will find that they won't match the colors that are on your display. Camera and printer manufacturers are getting better with the new Print Image Matching (PIM) feature - really just an extension of the EXIF metadata format, but the biggest problem is getting your monitor calibrated.


All in all, it would be great if usability considerations would be addressed in the digital camera workflow process. Things are getting better (iPhoto does a nice job by integrating steps 2-5 into a single application). However, it's really tough to compete with the film camera workflow. I can now understand why digital minilabs and dedicated 4 x 6 photo printers have a place in the marketplace.

Consistency == usability

One of the hardest things to deal with when learning something new is inconsistency. I rely on finding patterns in systems and reducing those patterns to idioms that I can easily remember. There have been two different things that I have been working on recently that have driven me crazy and in both cases the root of the problem is inconsistency.


OS X applications lack consistency in the way they handle the keyboard. It seems like every application has a different set of keystrokes for extremely common operations. By common operations, I mean things that devs do every day: next word, previous word, beginning of line, end of line, top of document, bottom of document. Among the three editors that I use on a daily basis, email and two different programming editors, they all have different keystrokes for these simple operations. In one editor command left takes me to the beginning of a line, in another editor it takes me to the previous word. Because of this, my TiBook just feels ... awkward.


In the other case, I've been learning a new API. This API is incredibly inconsistent. The abstractions break down in so many different places. Just when you think you've got the API figured out, it throws yet another curveball at you. It makes me appreciate just how difficult it is to create an API that is actually usable, especially when it has to evolve over several product releases, and you have to maintain backward compatibility.

Keynote

My copy of Keynote arrived today. I'm amazed that I ordered this software without even seeing a demo of it. So in some ways I was unsure of what to expect today. I have to report, however, that I'm totally blown away.


I'm a long-time PowerPoint user. Since I pay my mortgage by teaching software developers, PowerPoint has been a central part of my life for a long time. I thought that there wasn't much left to do in the presentation space, and apparently so did Microsoft, since there hasn't been much in the way of new features in PowerPoint in a long time.


So when I got my copy of Keynote, I fired up the demo slide deck that ships with the product. I was totally amazed by the eye candy that awaited me. There were beautifully done 3D transitions, as well as incredibly fast alpha blending cross-fades and animation. The speed with which my TiBook could render a picture twirling in from stage left was breathtaking.


Now I realize that with great power comes great responsibility. I'm sure that these effects will be abused by a great many presenters (myself included) for a while before we calm down. But I think that if used creatively, these things will add quite a lot of sizzle to a presentation.


The stock templates that ship with Keynote are really pretty as well. They reflect the rendering power and storage that is available on modern laptops. The presentations created by Keynote are not small by any stretch (Presentation Tips.key weighs in at a hefty 11.2MB), but honestly who cares?


Probably the most important feature of Keynote (and this was why I ordered it sight unseen) is the fact that Keynote stores its content as XML documents. Keynote documents are really directories, and within those directories you will find all of the media files associated with your presentation, as well as the content of your presentation in a presentation.apxl file.


The cool thing about having my content in XML is the fact that I can auto-generate my slide decks from other sources. One example is extracting code frags from actual source code files, rather than have to manually cut and paste that code into my presentation software.


So far, I'm very happy with Keynote. I'll post more observations once I get more time to work with it.

Cool Mac and Windows utilities

There are a number of really cool utilities that I really can't live without, and I thought I'd gather them here to generate a bit of discussion. In the Windows corner:


Dave's Quick Search Deskbar. This is by far the coolest utility that I have ever used. I haven't used anywhere near all of its features, but here's a list of the ones that I use often:



  • Typing in arbitrary text will search for that text using Google
  • amaz search terms /books will search for the search terms in the books section of Amazon
  • fedex tracking number will check the status of a fedex shipment
  • radix number will display that number in octal, hex, and binary
  • 0xhex= will convert a hex number to decimal
  • num1 op num2= will act as an inline calculator

There's tons of other features that I haven't used yet. Best of all, it's free!


Popup Stopper. I can't surf without this free utility that effectively blocks popup ads from appearing. However, there are times when it blocks legitimate applications, so you need to remember to turn it on and off as well.


Windows command completion. This lets you complete directory paths / filenames by pressing the tab key in a command shell. For example you can type "c:\Pro" and then hit the tab key to get "c:\Program Files" in the command prompt. This feature is enabled by default in fresh installs of Windows XP. For those of you who don't have this feature turned on, use regedit to set HKLM\Software\Microsoft\Command Processor\CompletionChar=0x9.


In the Mac corner:


LaunchBar: This utility lets you use the keyboard to launch applications and open Finder windows. For example, you can type "CMD-Space itu ENTER" to launch iTunes or "CMD-Space /L ENTER" to navigate to open a Finder window for the /Library directory. This saves me a lot of time every day.


MenuMeters: This utility places icons in your Menu bar that shows network activity, disk I/O, memory usage and cpu usage. It's great to be able to see what your computer is up to at all times in a nice and compact UI.


Since I'm new to the Mac world, I'd love to hear about more utilities that you can't live without in the Mac world.

The slippery slope of WORA

On my drive over to the Sun Tech Days presentation in Toronto, an idea kept rattling around in my head: Sun cares a lot about WORA (Write Once Run Anywhere). During the sessions that I attended, one of the things that Sun speakers stressed over and over again was the need for portability across different EJB containers. The question that I don't see addressed very often is: why does Sun care about portability?


The answer to this question isn't immediately obvious. After all, Sun is a hardware company that doesn't appear to make money directly from its Java technology suite. Java, however, is the technology that brought Sun into the forefront of public / developer consciousness. So I believe the question really is one of how Sun can turn Java mind share into hardware profits.


The answer to this question is that Sun wants Java applications to run on Sun hardware. In order to make that a reality, Java must be portable, otherwise there would be too much friction involved in migrating an application from one hardware / container platform to Sun's. By adopting this strategy, they have effectively tied the future of Sun to the idea that folks could develop Java code on their existing hardware, but that eventually, someday, those folks would reach into their checkbook and buy some Sun hardware.


While this is not necessarily a bad strategy, I think its a horrible strategy for Sun. Java portability is a double-edged sword that lets folks switch to Sun hardware, but it also makes it possible for folks to switch away from Sun hardware. The only way that this strategy can work effectively in the long term is if Sun can outperform the competition in price / performance.


I really don't see how this is possible, as Sun builds and sells proprietary hardware. Their hardware commands a considerable price premium over competitor's products today. In fact, no Sun hardware can be found in the top 10 results for TPC-C raw performance or price/performance. This premium is likely to continue to exist in the future because they lack the economies of scale that are available to the competition. Unless Sun can turn this around, it is likely that businesses will exploit Java's portability to switch away from Sun hardware.


As applications continue to migrate towards Web Services, things will only get worse for Sun since Web Services are yet another technology that make it easier for businesses to switch to the platform that provides best price / performance. It is unclear what that platform might be long term, but Sun faces some really serious competition in that space.

I went to the Sun Tech Days presentation in Toronto this morning to hear James Gosling speak. I had always wanted to see James speak, as I had always respected his contributions to software development through his Java efforts. I was horribly disappointed by what I heard.


Instead of hearing something insightful about the state of Java, we were subjected to a 40 minute rambling talk about nothing in particular. There were examples of Java being used in inconsequential applications - cel phone pets in Japan, automotive computers, and the Java application he was using to show his presentation slides. There was discussion about tangential topics: Moore's Law, Metcalf's Law, and how many developers were using Java. There was even some tangential Web Services bashing - "the amount of XML/SOAP traffic on your network is below detectable limits", which of course is surprising since the follow-on session was a description of Sun's Web Services strategy.


Gosling's talk was doomed from the start; he didn't have a story that he was trying to tell the audience. Instead, we listened to a number of totally unrelated anecdotes from his professional life. The only connection between those anecdotes was that Java was somehow involved in each one of them. At the end of the presentation, it wasn't clear at all to me why I should be excited by Java.


It's sad that Sun has to capitalize on his reputation in order to get an audience into a room. If this was a typical Gosling presentation, they really should keep him in the research lab and away from the public. The only cool thing about James was that he used a TiBook in his presentation :)


The tag-team Web Services speakers that followed James were even worse. The Web Services title of their talk was a smokescreen for their marketing message: why you need the Sun ONE developer platform. They talked for a couple of minutes about Web Services, but the rest of the time was spent as an infomercial for various Sun products. They even spent took a timeout to bash Microsoft by claiming that Java was more secure than the CLR since the speaker's mom could download and run Java applets on her computer!


The one redeeming thing about sitting through this presentation was that I had an opportunity to write this blog entry on my TiBook. Has John gone off the deep end?

Sam Gentile thinks that I might have gone off the deep end. First I run off and buy a Mac, next I decide that I need to create my own software commune. I'm fairly certain that Sam thinks I've become some kind of software hippy. He wonders how the Mac is going to impact my bottom line. That's a fair question; I'll try to address it in this entry.


As far as the bottom line is concerned, I've never been particularly motivated by money. That doesn't mean that I would turn down money that was offered to me, nor does it mean that I'm inexpensive to hire (I'm not). It simply means that the choices that I make are not primarily motivated by money. I've got more than enough money to maintain my lifestyle, so the rest of my time is spent on self-actualization.


I firmly believe that I have spent too much time being influenced by one school of thought - the Windows / COM / C++ school of thought. I feel that this school colors into my thinking in certain ways. Therefore, I have identified two areas that I want to investigate further. The first is how the Win32 API compares to Mac OS X as a client application development environment, and the second is whether dynamically typed languages really are more productive than the statically typed languages that I'm used to.


Mac OS X lets me spend time learning from folks who have a different solution to the same sets of problems that we try to solve in the Windows world. There are some really nice features in Cocoa, such as native rendering of documents in PostScript and a model-view-controller approach to building UI's.


Objective C has some elements of a dynamically typed language. I can send messages to arbitrary objects without knowing their types at compile time. This is a very different approach to programming than I'm used to, and it's great to see things from a different point of view. It's even more useful when you gain enough experience to be able to intelligently compare it with C++ and C#.[1]


A nice side effect of these investigations is that I am able to observe how I learn a new set of technologies. I think this feedback is extremely valuable in helping me improve my course materials for my Programming ASP.NET class. Too often, I forget what it is like to learn an entirely new technology like ASP.NET. These days I spend lots of time thinking about how to improve the learning process for my students.


My hope is that I become a better programmer after exploring these new horizons. Yes, I'm aware of the opportunity cost associated with these efforts, but I firmly believe that the opportunity cost of not investigating these technologies at this point in my career would be far, far greater.


[1] I don't think that I'm there yet, but I'm collecting a set of observations that I intend to write up in the future.

Observations on learning Cocoa

In between social engagements for the holidays, I've managed to finish James Duncan Davidson's Learning Cocoa with Objective C book.


It's a good first book on writing Cocoa applications, and will get you familiarized with Project Builder, Interface Builder and Objective C. It's real strength lies in the fact that there are lots of "labs" in this book. Each chapter contains a self-contained application that you build by typing in code or following step-by-step instructions where Interface Builder is involved. This gives you a real sense of accomplishment and builds confidence as you tackle progressively more difficult projects.


It's been such a long time since I've learned a new technology this way. When I was first learning C# / FCL, most of that involved writing command-line applications that explored various classes in the libraries. I have fond memories of dumping stack traces to the console window once I discovered the existence of the System.Diagnostics.StackTrace class.


These two approaches are dramatically different from each other. In the case of the CLR, I approached learning technologies from the "bottom up". It was a long time before I looked at high-level technologies in the stack such as ASP.NET or Windows Forms. In the case of Cocoa, I'm definitely starting from the "top down". I'm finding my way around an IDE (there was no IDE in the early days of the CLR), learning a new programming language, and learning a new class library.


I suspect the main reason why I am approaching learning OS X the way I am is that I don't know how the underlying OS works. I don't understand how code is packaged for execution, how it's loaded, how processes are structured etc. I took for granted all of that knowledge when I approached learning the CLR for the first time. During that experience, I spent a lot of time thinking about "well Win32 does it this way, I wonder how the CLR does it?"


What I do know is that I need to be able to write some code to write test programs as I explore the frameworks. Since console applications are so 1970's, I decided to learn enough about writing GUI applications using Cocoa to be dangerous. Once that has been established, I can revert to my time-honored techniques for spelunking in new territory. To be honest, I really should be writing console applications and exploring things from the ground up, but I just can't resist that UI :)


Now that I've finished my first Cocoa book, I'm working my way through my second one: Simson Garfinkel and Michael Mahoney's Building Cocoa Applications: A step-by-step guide. This is a much more detailed book than Davidson's. In particular, Chapter 4 shows how to build a Cocoa application without using Interface Builder. This is extremely useful, since IB is a giant black box; it's so important to be able to understand what IB generates for you.


A quick note: IB differs from Windows frameworks such as MFC or Windows Forms in that it doesn't generate code. Instead, it appears to create various Cocoa framework objects at design time to express your user interface. Then when you are ready to write code, it serializes that object graph into a nib (NEXT Interface Builder) file that is then consumed by your Cocoa application. A nib to XML deserializer would be a really fun tool to write :)

Interface Builder Rocks!

Interface Builder is an awesome forms designer package. I really like the way it understands the Aqua Human Interface Guidelines. For example, if you drag a UI element like a text field close to the edge of the form, IB will draw a dotted blue line that tells you where the boundary lies. This makes it really hard to create form-based applications that don't look "Mac like". Another cool feature happens when you select a UI element, hold the option button down, and roll your mouse pointer over another UI element. Serialization

There's some interesting XML serialization functionality built into Cocoa. You can write some fairly simple code such as:


NSMutableArray *array = [[NSMutableArray alloc] init];
[array addObject:@"John"];
[array addObject:@"Carolyn"];
if( [array writeToFile:@"people.plist" atomically:YES] )
  NSLog( "succeeded" );


which results in the following XML being produced:


...
<array>
  <string>John</string>
  <string>Carolyn</string>
</array>


Apparently Mac OS X uses XML configuration files known as plist files to configure functionality. There is a simple tool called the Property List Editor for editing these files.


I don't yet know how to serialize arbitrary objects (I tried serializing an NSArray of ABPerson objects, and that didn't work). Unfortunately, writeToFile failed silently without giving me a clue about what was causing it to fail.


The one thing that I am missing in Cocoa so far is a stream-based I/O model. It seems a bit crufty to have to send messages like writeToFile instead of writeToStream.

IUnknown lives!

This morning, I found myself typing strangely familiar code:


- (void) setName: (NSString *) name
{
  [ name retain ];
  [ name_ release ];
  name_ = name;
}


My first few minutes programming with Objective C were fairly painless. Surprisingly, most of my C/C++ instincts worked just fine. My really rusty COM instincts kicked in when I saw the retain (aka AddRef) and release messages. I was surprised that my C# instincts worked fine too - I was instinctively NOT typing semicolons at the end of the message sending operators (those scary looking [ ... ] things in the middle of the code), which of course the compiler gently reminded me of when I tried to build my project.


However, after not programming in C++ for quite a few months, I was revulsed by the fact that I needed to have both an interface and an implementation section.
