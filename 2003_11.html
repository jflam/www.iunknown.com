Algorithm complexity and modern CPU's

Remember your college introduction to data structures and algorithms course? Remember how you spent some time calculating the complexity cost of various algorithms based on big-O notation? There was one very important assumption that was made whenever you made those calculations: that the cost of accessing any address in memory was constant.

With today's multi-tiered CPU cache architectures, it has become increasingly obvious (to those who care) that that assumption is no longer valid. A full cache miss on a modern 3GHz+ P4 will mean that you will waste ~1000 operations while waiting for the data to show up at the CPU.

Jan Gray wrote an excellent essay that attacks a data structure that is responsible for lousy performance characteristics on modern CPU's: the venerable linked list. In this essay, you'll see some of Jan's prognostications on the future of CPU architectures. My favorite quote from this piece is "This scenario is so bad and so common that the microprocessor vendors use 80% of their transistor budgets for on-chip caches -- Intel as glorified SRAM vendor.".

C++ programmer ethos

Why is it that whenever I start writing [Managed]C++ code that I:

November 06, 2003

Assembly Friendship

[Via Charles Cook] Jim Hogg from the CLR team talks about an upcoming new feature in V2 of the CLR: "Friend Assemblies". There will be a new assembly-level attribute called InternalsVisibleTo that lets the author of an assembly specify who it wants its internal methods to be visible to.

This is extremely useful for unit-testing. There are many occasions where you need to make the internals of a class visible to unit-testing code. Today, this essentially forces you to have to conditionally compile your unit-testing code into the same assembly as the code that it tests. In V2, you will be able to cleanly separate the code from its tests. It also enables a scenario where InternalsVisibleTo can be used to allow diagnostic code to run against a shipping assembly at a customer site.

November 05, 2003

Ant community reaction to MSBuild

Steve Loughran posts his initial thoughts based on a reading of the docs. He has a number of interesting criticisms of MSBuild, particularly about pushing dependency analysis to the target level. Steve is a co-author of the excellent book Java Development with Ant and an Ant committer.

Stefan Bodewig posts several entries about MSBuild.

In target depenencies Stefan raises some interesting issues about dependency analysis, much like what Steve did in his entry. This is a particularly interesting problem when you consider what happens with dependencies if you want to incrementally update an FTP server. It's unclear to me how Inputs/Outputs would conflict with an FTP task that did smart dependency analysis against remote date/time stamps.

In conditions in MSBuild and Ant Stefan advocates using properties files in Ant over XML property elements. The problem I have with property files in general is that they aren't XML. This makes it much harder to manipulate these files through generalized XML techniques such as XSLT. It's not impossible, but it increases the friction considerably. +1 to MSBuild for avoiding property files (even at the cost of additional verbosity).

In Items in MSBuild Stefan examines the way tasks can be decoupled via Items. Specifically, it's the OutputItem element in MSBuild that allows tasks to both consume and emit Items, which makes it far easier to couple the output of one task to the input of another. He also proposes a very interesting "pipe" syntax that would allow tasks to be decoupled without incurring the friction involved in defining an OutputItem for a task.

MSBuild on v1.1.4322

I just tried adding the following app.config file to msbuild:


<?xml version ="1.0"?>
<configuration>
    <startup>
        <requiredRuntime safemode="true" imageVersion="v1.1.4322" version="v1.1.4322"/>
    </startup>
</configuration>

It seems to work ... so far. That's pretty good news if you want to try out MSBuild on your existing V1.1 projects. I haven't tried this yet on a machine that doesn't have V1.2 installed yet - any takers?

Update: It doesn't work. I just tried it on a V1.1 framework box. There is at least one dependency on V1.2, but it appears to only occur on an error path in the code: V1.2 defines a SecurityException.get_PermissionStateInstance() method that isn't present in V1.1. No time to dig through why the security exception was thrown in the first place, though ... I guess we'll have to wait and see whether there's an "official" implementation in the future from MS.
