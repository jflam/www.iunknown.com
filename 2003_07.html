UI Programming and Unit Tests

I spent some time today thinking about the best way of writing unit tests for GUI code. The key is determining whether your rendering code is issuing the correct set / sequence of commands to the graphics subsystem.

At some point, you have to take it on faith that the commands that you issue to the graphics subsystem will result in pixels being painted at the correct positions on the screen. This is not what I'm trying to accomplish in my unit tests; that's what Microsoft and the device driver writers have to guarantee will happen.

Instead, I would be happy if I could capture a "correct result" in my unit tests. Imagine that you could describe the correct result as an XML document that contains the rendering commands and their parameters. For example:

<DrawEllipse>
  <Pen name="Pens.Black"/>
  <Rectangle x="1" y="1" width="100" height="100"/> 
</DrawEllipse>

Next, imagine that I can intercept the call to Graphics.DrawEllipse( Pen pen, Rectangle rect ). I could then log the method call as XML, which can easily be compared to the correct result using an XML diff tool.

In practice, this is a lot harder than it looks. If you examine the System.Drawing.Graphics class, you'll find two problems. The first is that the class is sealed, which prevents you from deriving a plug-compatible class from it. However, even if the class wasn't sealed, you would wouldn't be able to intercept the methods because methods like DrawEllipse aren't virtual.

In an ideal world, Microsoft would have defined the interface to the Graphics class via an IGraphics interface. This way, we would be free to cruft up our own plug-compatible implementation that could be used in place of the real Graphics class when we run our unit tests.

One way of solving this problem would be to pull out the ultimate hacker weapons - runtime method interception extensions to the CLR. My project-in-limbo, CLAW, would have let me solve this problem by defining an interception layer between the caller and the real Graphics object.

There is, of course, a simpler way of solving a subset of the UI rendering problem. You can create a Graphics object that wraps the existing Graphics object. 

public void DrawEllipse( Pen pen, Rectangle rect ) {
  if( Test ) 
    LogCall( pen, rect );
  else
    Graphics.DrawEllipse( pen, rect );
}

The problem with this approach is, of course, that this only works with code that was compiled explicitly against the surrogate Graphics class. So this technique would only be able to test your custom rendering code, but wouldn't be able to test whether Framework rendering code was rendering correctly.

Any thoughts about how to solve this problem without resorting to runtime method interception of the real Graphics class would be welcome.Â

Freemind: an incredible thought organization tool

I've been using an open-source tool called FreeMind for the past couple of days. This is hands-down the most incredible brainstorming / thought organization tool that I've ever seen.

FreeMind is essentially a fancy XML editor. It lets you create single-rooted recursive hierarchies of information. But the presentation and editing is so powerful and intuitive that it pretty much eliminates any friction involved in restructuring your XML document. You also have the ability to link to external URL's and external files.

The way I use FreeMind is to initially capture a mind-dump of ideas relating to a single topic. Then I start noticing clusters of ideas that relate to one another and I use FreeMind to reorganize those ideas into their natural order.

See for yourself. Surf over to the FreeMind homepage on SourceForge. Make sure that you check out the screenshots of the program. It was worth reinstalling the Java runtime on my computer just to run this application.

How to misuse code coverage

Steve Makofsky Steve points to this excellent paper by Brian Marick that focuses on an important source of bugs: faults of omission. Your code coverage tool can only tell you about code that exists that wasn't executed. It can't tell you anything about code that you need to write - that is code that fixes faults of omission. The simple rule of thumb that Brian proposes is that:

If part of your test suite is weak in a way coverage can detect, it's likely also weak in a way that coverage can't detect.

I strongly recommend that all developers take a close look at Brian's article. I had to re-read his article a few times to get what he's saying, since his recommendations are quite abstract. But you won't regret the time invested.

I'm now thinking about integrating a code coverage tool into my NAnt/NUnit build system. There doesn't seem to be any open source CLR tools yet - I'm going to take a quick look at Rational's Quantify and NuMega's DevPartner products to see whether they can be integrated into an automated build system.

What does "Enterprise" mean?

The term "Enterprise" seems to have many meanings. Here's my stab:

  • Enterprise applications are much more expensive
  • Enterprise applications are used by more than one person concurrently
  • Enterprise applications are typically browser-hosted today
  • Enterprise applications solve business problems1
  • Enterprise developers need to use more expensive tools

But I don't think that Enterprise applications are necessarily any more complex than non-Enterprise applications. I don't think that anyone would call a C++ compiler an Enterprise application, yet I would argue that it is far more complex than most Enterprise applications today.

From a developer standpoint, do you really need to use Enterprise tools (read: more expensive) to build Enterprise applications? I use VS.NET Enterprise Architect because it's the most expensive version of VS.NET that comes "for free" with my MSDN Universal subscription. I know lots of folks who install Windows 2003 Enterprise Server because itâ€™s the most expensive version of Windows (I know I did). I know it makes us feel better to use the more expensive tools, but do we really need to use any of the more expensive features?

The reason why I'm writing this is due to my frustration with tool vendors that only seek to sell expensive bundles of software to "Enterprise" developers. As an example, I can't buy a copy of NuMega's SoftICE debugger without buying their SoftICE Driver Suite product which costs $999!. Now, I don't write device drivers, but I could use a proven kernel-mode debugger every now and then. NuMega has lost me as a potential customer, and I'm thankful that MS gives away WinDbg for free for the odd time that I need to use a kernel-mode debugger.

[1] As opposed to technology problems, although you can argue that some business problems ARE technology problems.

The truth about cscomp.dll

On the DOTNET-LANGUAGE-DEVS mailing list, I posted a question about why there was a cscomp.dll even though the frameworks don't use it directly. I just double-checked today, and the only folks that use cscomp.dll is VS.NET. You can verify this for yourself by running PROCEXP on your computer.

The ASP.NET folks elected not to use cscomp.dll so that they could sandbox the compilation process. Here's the answer from Grant Richins at MS (via Lars Bergstrom):

"Cscomp.dll is loaded in-process for the IDE scenario. He is correct that avoiding the call to CreateProcess would save some time, but by running the compiler out-of-process, ASP.NET is able to isolate itself even better by giving the compiler process very few privileges (at one point the compiler couldn't even write to the temp directory!). Also any exception that happens in the compiler is completely isolated to the compiler process and has no way to bring down all of ASP.NET.


We had considered exposing a managed in-process C# compiler for ASP.NET and the CodeDom. In fact we even wrote the code (see Microsoft.CSharp.Compiler), but when ASP.NET and the CodeDom decided not to use it, we decided to deprecate it. Its only real savings (besides the unneeded CreateProcess) was that it could compile source code passed in as strings and thus not cause file I/O, but I'm not sure if that savings wasn't swallowed up by the cost of interop. We wanted to remove it entirely because we didn't have good test coverage for it, but too late in our ship cycle we discovered some other teams had introduced dependencies on this interface unbeknownst to us. So we recommend customers don't use it because we are planning on ripping it out in a few versions."

I just noticed a cool feature in Reflector's decompiler

While spelunking around inside ASP.NET tonight, I noticed that Lutz has added some more intelligence to his decompiler. The CLR doesn't preserve local variable names in metadata, so decompilers need to assign some kind of symbolic name to the variable. If you look at tools like ILDASM, it uses rather unimaginative names like V_0, V_1, etc. Reflector, on the other hand, takes the last "word" in camel-cased notation and uses that as the variable name. For example, IDictionaryEnumerator becomes enumerator1. What a brilliant idea! Well, done, Lutz!

Improving OOBE for open source?

The CodeRef build environment uses three different open source tools: NAnt, NUnit, and NSIS. One of my challenges is improving the Out-Of-the-Box-Experience (OOBE) for folks who want to work on the source code. I really want to avoid the long list of instructions for creating the build environment, since I firmly believe that this discourages people from even trying to work on the source code. Instead, I would like to have my build script automatically install the correct versions of those tools in a private path that gets setup for just that project.

Currently, I'm packaging a minimal set of bits from NAnt, NUnit, and NSIS along with CodeRef. This worries me because I'm not sure whether I'm a) violating the license agreement(s) by repackaging the bits, and b) whether the bits that I pull out will actually work reliably in my build environment.

To improve the OOBE, the various open source tools in the .NET world need to standardize on an installer that can be scripted from the outside. This way a master install script could install NUnit 2.0 and NAnt 1.0 for one project, and NUnit 3.0B2 and NAnt 0.8.3RC1 for another project. p. The way I envision this working is as follows: a "master" installer dictates to the "component" installer what directory tree to install into. The component installer has to report the "bin" path of the installed component to the master installer. The master installer would use the path information to create a shell script that would setup the correct build environment. This is similar to the way the vcvars32.bat file sets up the command-line environment in VS.NET.

Do folks have thoughts along these lines? I'm fairly certain that NSIS can't do what I need, and I don't have enough knowledge about MSI's to know whether they can fit the bill.

Foiled by csc.exe

I was writing a NAnt build script yesterday that compiles a number of separate projects. Some of these projects are really simple, and I felt that it was a waste to have to create a new Win32 process to host csc.exe as it compiled the code. So today, I spent some time poking around with Reflector to see whether the System.CodeDom classes would let me host the C# compiler inside of the NAnt Win32 process.

Unfortunately, this does not seem to be the case. If you decompile System.CodeDom.Compiler.Compile() using Reflector, you'll see that it simply shells out to csc.exe via the call to Executor.ExecWaitWithCapture(). This is somewhat disappointing since csc.exe is simply a wrapper around cscomp.dll, which exports a method called CreateCompilerFactory() which is called from csc.exe.

I'm surprised that MS didn't choose to use cscomp.dll from Microsoft.CSharp.CSharpCodeGenerator. I would imagine that this would give ASP.NET a significant performance boost when compiling .aspx and related files since this avoids the overhead involved in calling CreateProcess. I'll have to hunt around a bit more in System.Web.dll to make sure that this isn't in fact the case.

Framework 1.1 and NUnit gotcha

If you're using V1.1.4322 of the frameworks, and you're writing unit tests using NUnit 2.0, you'll need to add the following to both the nunit-console.exe.config and nunit-gui.exe.config files: 

<startup>
  <supportedRuntime version="v1.1.4322" /> 
</startup>

NUnit 2.0 is compiled against V1.0 of the frameworks, and at some point you'll wind up trying to load both V1.0 and V1.1 versions of mscorlib.dll, which is prohibited by the CLR. By adding this to your application's configuration files, you'll force the NUnit test harness code to run under V1.1 of the frameworks, thereby solving your problem without forcing you to recompile the NUnit binaries.

Fun (sort of) with XML

Like Craig, I’ve been spending some time building an XML authoring system.  Yesterday, I wrote a simple WordML compiler that will take an existing document and insert external references into the document. My motivation for doing this was to try and eliminate the inevitable bugs that creep into a manuscript whenever you cut and paste code fragments into the document. What my system lets you do is insert a reference like:

<<\temp\somecodefile.cs>>

into your word document, and my WordML compiler will replace the reference with the contents of somecodefile.cs.

This works fine for my current writing project, “A Busy Developer’s Guide to GSLgen”. However, later on, I’m going to need to insert some code fragments into my Word document. This requires a bit more processing on my part, which I can accomplish with a bit of regular expression work. For example, if I had this reference:

<<\temp\somecodefile.cs#codefrag1>>

It would pull a code fragment that is marked using comments and #region statements, such as:

// #region codefrag1
public void DoSomething()
{
}
// #endregion codefrag1

Since I want this to work with more than one kind of source code file, I’m going to build a plug-in mechanism that allows folks to write their own regular expressions that parse source code files in their favorite programming language.

One other thing: I transform the WordML document in place via XmlDocument. I did this mostly because I was in a hurry and I didn't want to deal with the pain that is XmlReader/Writer. It would be so much nicer if there was sed-like support for modifying XML streams in the frameworks.

I’ll release the source code in the next couple of days once I’ve had a chance to dogfood it.

Another one bites the dust

I see that Clemens Vasters has also soured on AOP. The primary weakness of the AOP approach is due to the difficulty of writing stateful aspects that are composable. In his piece, Clemens calls stateful aspects "opaque aspects" and stateless aspects "passive aspects". While he describes a bunch of nasty technical issues that got in the way of having a clean implementation, the composability problem will nevertheless limit the utility of AOP.

The only point that I disagree with in his article is how he characterizes AspectJ pointcuts as equivalent to the use of attributes in the CLR. AspectJ pointcuts are an example of a "push" mechanism for weaving aspects, whereas attributes are an example of a "pull" mechanism for weaving aspects. In AspectJ, you don't need to touch the source code to weave an aspect into a particular join point, whereas with attributes you do need to touch the source code to achieve the same effect.
