Obscure metadata trivia

Posted by John Lam Wed, 19 Jul 2006 21:31:00 GMT

Is this a bug?

public enum HideEm {
  ToString,
  Equals
}

Well, the C# compiler doesn’t think so, and it doesn’t even issue a warning.

But what’s really going on here is that you’re defining static fields called Equals and ToString, which of course mask the definitions of the corresponding methods in Object.

Now, if you’re a statically typed language you don’t even care since the compiler can infer whether you’re referencing a field vs. calling a method. But in Ruby, how do you distinguish between these things?

More method overloading madness in RubyCLR

Posted by John Lam Wed, 19 Jul 2006 17:12:00 GMT

Yikes. This method overloading + shadow class inheritance problem turns out to be even harder than it looks. I suspect anyone writing code to interop with .NET are going to run straight into this problem.

Here’s a description of this new scenario:

class A {
  public int M1() { return 1; }
}

class B : A {
  public new int M1() { return 1; }
}

If you create an invoke M1 via an A object reference:

A.new.M1

You will have created the initial shim method on the A shadow class.

Now if you try to invoke M1 on B via:

B.new.M1

This will call via the M1 shim method on class A. Right now I rely on the method_missing calls to generate shim methods on demand. Clearly this can’t be the case for dispatching these kinds of non-virtual methods that can be redefined in derived classes.

It seems to me that the method shim would have to see whether the object reference being called is derived from the “most derived” class that the shim knows about. If not, it will have to redefine the method shim on the fly at runtime and bind it to the shadow class that the invocation is happening through.

Yuck. This is a LOT of code (and a fundamental redesign of the code since there are lots of assumptions baked in about this kind of behavior) to handle this corner case that I suspect will be pretty rare out there in the wild.

Comments welcome about the “most derived” algorithm would be welcome.

Shadow class inheritance and method overloading

Posted by John Lam Wed, 19 Jul 2006 15:54:00 GMT

Shadow class inheritance can break in RubyCLR due to method overloading. The problem has to do with redefining methods in derived classes and relating that back to the problem of Ruby not allowing method overloading.

Consider this set of C# classes:

class A {
  public int M1() { return 1;  }
  public int M1(int x) { return 2;  }
}

class B : A { }

class C : B {
  public new int M1() { return 3;  }
}

Here are some interesting ways of calling M1():

A a = new A();
B b = new B();
C c = new C();
Console.WriteLine("new A().M1() = {0}, new A().M1(1) = {1}", a.M1(), a.M1(1));
Console.WriteLine("new B().M1() = {0}, new B().M1(1) = {1}", b.M1(), b.M1(1));
Console.WriteLine("new C().M1() = {0}, new C().M1(1) = {1}", c.M1(), c.M1(1));
Console.WriteLine("(new C() as A).M1() = {0}", (c as A).M1());

This poses some design issues for how methods are dispatched in RubyCLR:

  1. The method shim (M1) is defined on A’s shadow class and calls M1() and M1(int). This method shim will be inherited by B’s shadow class.
  2. A new method shim (M1’) is defined on class C’s shadow class and calls C::M1() and A::M1(int).
  3. There must be some way to call A::M1() from a reference to an instance of C.

The code from above in Ruby would look like:

assert_equal 1, A.new.M1
assert_equal 2, A.new.M1(1)
assert_equal 1, B.new.M1
assert_equal 2, B.new.M1(1)
assert_equal 3, C.new.M1
assert_equal 2, C.new.M1(1)

Now the last case is the hard one syntactically -

assert_equal 1, C.new.M1_of_A

Would something like _of_(declaring class) as the name of the shim that calls the correct method the right way of doing this?

Comments appreciated!

RubyCLR update

Posted by John Lam Wed, 19 Jul 2006 13:54:00 GMT

It’s been a long time since I’ve updated you on the status of RubyCLR.

I’ve had some cycles this week to start working on squashing bugs in RubyCLR. The nasty one was due to my bone-headed implementation of the proxy object cache. Bottom line: I relied on Object.GetHashCode() to provide process-wide unique identifiers which, of course, it does not. Frankly I’m rather embarrassed that I made this bug.

I have fixed my object cache to use GetHashCode() to assign objects to buckets, but I do a manual Object.Equals(a, b) comparison as I walk the bucket list to look up the appropriate CLR proxy object. So I now handle identity correctly. Note that identity is still wierd with respect to interfaces, but that’s going to be a longer-term fix.

I have also implemented shadow class inheritance. This lets me add helper methods to base shadow classes that propagate to derived shadow classes. This lets me be much more efficient in memory since method shims exist exactly once in an inheritance hierarchy whereas they used to be duplicated. So this lets me add a lovely helper method for doing some WPF coding:

import System::Windows::FrameworkElement

class System::Windows::FrameworkElement
  def [](name)
    self.find_name(name)
  end
end

The import directive creates a shadow class object for FrameworkElement. I then define a method for the square bracket operator which delegates the call to the FindName method of FrameworkElement. This lets me do this from anywhere in my code:

w = Wpf.load('myxaml.xaml')
w['button1'].click do |sender, args|
  puts 'clicked me!'
end

Update: I’ve had a lot of problems trying to check in source code changes to RubyForge recently. I haven’t had the cycles to track down this problem, so I’ve created a new release for this code update. You can get it from here.

Upgrading my MacBook Pro HDD was fun

Posted by John Lam Thu, 13 Jul 2006 04:28:00 GMT

I upgraded my MacBook Pro HDD to a 100GB 7200rpm disk tonight. Before tonight I was using the 80GB 5400rpm disk that shipped with my computer. I was amazed by how straightforward the experience was.

I began by using Disk Utility to create a copy of my system partition onto an external USB 2.0 HDD. This took a long time – I let it run overnight. Are there better (faster) applications for creating an image of my system partition?

I then used the most excellent MacBook Pro Disassembly Guide to pull apart my MacBook Pro to physically replace the disk. One thing you shouldn’t do, however, is disconnect the keyboard and trackpad. I just propped the keyboard against the LCD while replacing the HDD.

I started my Mac and booted the OS from the DVD drive. I created a HFS+ partition on my new disk and installed Mac OS onto it. When I was finished, I plugged in my USB drive, and used the Migration Assistant to copy over my old user settings and files. This took about 1.5 hours and I had a machine that was every bit the same as my original!

The only app that I had to reinstall was Parallels.

What a nice experience. And the machine definitely feels much snappier as well.
