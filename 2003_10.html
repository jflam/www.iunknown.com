NX: No execute coming in Win XP SP2

Scoble just posted a link to the Developer's View of Windows XP SP2. It's worth reading the section on the new memory protection features in SP2, specifically the No-Execute (NX) flag. It requires hardware support from the CPU, specifically the ability to set a no-execute flag on each virtual page. This would effectively make it impossible to execute code that was injected via a buffer overrun attack on a data page. Currently the only CPU's that implement this feature are AMD's K8 and Intel Itanium processors, but "it is expected that future 32 and 64-bit processors will provide execution protection. Microsoft is preparing for and encouraging this trend by supporting execution protection in its flagship Windows operating systems."

What's new in the Whidbey Frameworks?

I spent yesterday writing a small utility that will diff two different sets of assemblies. So, if you're at the PDC, you can calculate the diff between V1.1 and V1.2 of the Frameworks. This can very rapidly give you a heads up about what has been added / removed / changed in the frameworks.

This is a rather different utility than most that you've likely run across. It is a utility that is implemented as a set of MSBuild tasks. There are two custom tasks that I implemented. The first will take a set of assemblies and dump the names of all of their types to a flat text file. The second will shell out to diff.exe (a GNUWin32 tool) to calculate the actual difference file.

To run the utility, you need to:

  1. Download and unzip the files
  2. Download and install the GNUWin32 diffutils if you don't have them installed already.
  3. Navigate to the AssemblyDiff\Test directory
  4. Run moveit.cmd, which will copy core.dll and tasks.dll to your V1.2 Frameworks directory (\Windows\Microsoft.Net\Frameworks\V1.2.30703\ on my computer).
  5. Type msbuild on the command prompt.
  6. Examine the generated diff.txt file in the directory.

In the meantime, enjoy the utility. Feedback and comments are more than welcome!

Some hints for y'all at the PDC

I am so bummed that I can't be there with y'all this year. My new family has to come first, so I'm staying at home this week. I'm living vicariously through your blog entries, so keep them coming! Here's a few tips for the software that you'll be receiving in your goodie bags today/tomorrow:

First, make sure that you have a copy of VMWare / Virtual PC installed. You should have a fresh copy of WinXP / Win2K3 setup already as a baseline image. You should install the Whidbey bits onto that VM. If you're using VMWare, you should change the VM preferences so that CTRL-ALT-SHIFT is the VMWare hotkey (it defaults to CTRL-ALT). This is particularly useful since the VMWare hotkey sequence is also part of the key sequenceÂ that brings up online help in VS.NET (e.g. CTRL-ALT-F2 for the Index). You'll be much more productive once you have that up and running. Also remember that to login you'll need to type CTRL-ALT-SHIFT-INSERT to send a CTRL-ALT-DEL to VMWare.

Next, if you're planning on installing the Longhorn build, you'll be generally better off installing it onto a separate partition on the fastest computer (and fastest video card) that you have. I'd highly recommend a minimum of 64MB VRAM on your video card. This is also generally the biggest baddest video card that you can get on a laptop these days. You'll be even better off just buying another HDD for your laptop and swapping it in and out to run LH. There are also widespread reports of LH running on Virtual PC, but you won't get the full experience if you run it in a VM.

Finally, you'll probably be wondering what has changed between version 1.1 and 1.2 of the frameworks. I'll be posting a little app here tomorrow that will show you the differences. It's written using 1.2, so you'll need to have those bits installed to run it.

Enjoy yourselves!

In defense of PDF ...

Wow. I sure touched a nerve with my previous post on what I think blog entries should be. In reality, what I wanted to do was throw out the idea of using PDF as a way of controlling the format for longer blog pieces.

Jon Flanders, an old friend from my time at DevelopMentor, sent in a link to a Jakob Nielson article on avoiding PDF for on-screen reading. It's an interesting piece, and reflects much of the frustration that folks have with the current state of PDF on web sites today.

I'm about to launch a newsletter later today that will be distributed in PDF form. Here are my reasons why I chose PDF as the medium:

  1. I like to read my content one page at a time, much like I would in a traditional book or magazine.
  2. I want to include some formatting in my newsletter to help increase the readability of my content.
  3. I want the entire article to be available without additional roundtrips to the server. I hate articles that force me to read pages one at a time, but require a roundtrip to the server to retrieve the next page (almost all of the professional news/review sites are like this).
  4. I want the content to be available offline for folks who want to read that way. I find that I tend to carve out reading time when my laptop is not connected to a network somewhere.
  5. I want folks to be able to meet the above requirements and still be able to forward my newsletter around as an email attachment. This can only be accomplished if all of the content resides in a single file.

To me, PDF is the only medium that meets all of these requirements. And as long as I format my content so that it can be easily read on-screen (respecting the 4:3 layout of most monitors and using a double-column layout) I think I won't upset too many people. You'll find out later on this afternoon, I promise.

VB.NET and NAnt gotcha

Today, I wanted to add some VB.NET code to an existing solution where all of the other projects were written using C#. I had a working NAnt build script for this solution, so I thought it would be a piece of cake to add a target to my build script to build my VB.NET assembly. Boy was I wrong. An hour later, I figured out the answer.

VB.NET does a couple of things to make your life easier. First, it imports a number of commonly used namespaces on a project-wide basis. On my computer, VB.NET automatically imports the following namespaces: Microsoft.VisualBasic, System, System.Collections, System.Data, System.Diagnostics. Second, it assigns a "root namespace" to all types defined in the assembly. The "root namespace" defaults to the name of the project. You can see these settings under Project|Properties in Visual Studio .Net. The interesting tabs are Common Properties|General and Common Properties|Improts.

While these are nice features which save you time, it can really make a mess of your NAnt build scripts which don't have the same set of default imports. Fortunately, the vbc task in NAnt defines attributes that you can use to define these settings. The first is the imports attribute where you can add the appropriate default namespace imports, as shown below:

<vbc ... imports="Microsoft.VisualBasic, System"/>

The second is the rootnamespace attribute where you can define the default namespace for the project, as shown below:

<vbc ... rootnamespace="MySolution.MyProject"/>

I hope this saves someone else out there some time.

Stefan Bodewig on understanding open source licenses

The NAnt community is contemplating changing the license of NAnt to a less restrictive license (it's currently GPL). As part of some of the discussions on the mailing list, Stevan Bodewig (a long-time Ant contributor and member of the Apache Software Foundation) chimed in with his analysis of the differences between the major licensing models: MIT, BSD, ASL, and GPL. Enjoy.

Trying to chime in with a bit of experience from seeing project migrate to the ASF.

On Wed, 8 Oct 2003, Gert Driesen wrote:
> From: "Clayton Harbour"

>> On another note you mentioned license changes, I have always been
>> curious how that works/ is accomplished?

If you want to be really really save: Get all people who have ever contributed anything to your code to agree on changing the license with the next release.

You cannot change the license for your older releases, they've already been out. And the people who have contributed code still hold the copyright to it (even under US law AFAIU).

Doing it with the next release also gives people who prefer the old license the opportunity to have a stable forking point. They can pick up the code from before the license change and create a new project from there.

> I did have a quick look at the licensing stuff, and to me it seems
> like a BSD-style license is the most open license ...

I feel I should put a big disclaimer in before I proceed. I'm a member of the Apache Software Foundation. I am biased. I'll try to be fair when comparing the licenses, but may very well fail to do so.

All major open source licenses have their pros and cons. The (L)GPL on the one side and the BSDish licenses on the other simply have a different focus.

Behind the (L)GPL is the philosophy of Free Software as defined by the FSF. It is a philosophy and a political manifesto that you can agree with or not. The GPL has been crafted to enforce this vision.

The GPL as well as BSDish licenses give their users the right to get the code for free and to modify it. The GPL takes away the user's right to distribute the modified software under different licensing terms, and it does so because it wouldn't be "Free Software" anymore otherwise.

If you have any problem with big companies making money with your software without giving anything back to you - the BSDish licenses are not for you.

As for "most open", I'd say the MIT license1 probably is the winner here. It basically says "do with this what you want, don't blame me".

The BSD license2 adds one thing "don't advertize with my name, at least not without asking for permission".

And the Apache Software License3 adds brand protection on top of this. "Don't call your derived project like mine".

Note that this brand protection is the part that makes the FSF claim the Apache Software License was incompatible with the GPL (i.e. a GPLed software must not include Apache licensed code).

Footnotes:
1 http://www.opensource.org/licenses/mit-license.php

[2] http://www.opensource.org/licenses/bsd-license.php

[3] http://www.opensource.org/licenses/apachepl.php

The rest is my personal opinion, don't read on if you don't want to see that 8-)

I for one don't agree with the FSF's vision, I feed my children from money I get for writing commercial software and I don't feel bad about it.

I enjoy writing software and giving it away for free as well - and if I contribute to a open source project that uses a BSDish license I get the additional benefit that I may use my own code (plus improvements made by others) for the commercial software I write.

If a big company comes and takes my code, modifies it and redistributes it as a commercial product, this is no problem for me.
I don't believe in "free software", I believe in improved software and that the commercial company will realize that it is in their better interest to contribute their changes back instead of merging my changes with theirs over and over again over time.

Stefan

pslist and pskill to the rescue!

Occasionally, explorer.exe locks up hard on my desktop that's running Win XP Pro. This makes it hard to do virtually anything that uses a GUI, including using task manager to shut down the offending process. However, I ALWAYS have a console window up, which uses a completely different UI. Therefore, it is immune to lockups caused by a badly behaving explorer.exe process.

If this happens to you, there are two utilities from SysInternals that come to the rescue: pslist and pskill. pslist lets you view a list of all running processes on your system, and pskill lets you kill a selected process. pskill even lets you kill named processes, so you can type "pskill explorer", and it will kill all named instances. These utilities also run just fine on a non-admin account :)

The magic ratio 4:3

I've been thinking about writing a newsletter for quite some time now. There is one thing that has really frustrated me about everybody else's newsletters (and web sites - this one included). They are designed based on the assumption that you have a portrait monitor (e.g. 3:4). Now, almost nobody has a monitor oriented this way, which explains why many web sites look rather goofy when viewed maximized on a 4:3 monitor. Most news sites, like the New York Times look like they were designed to be printed out, not viewed on a monitor.

I'm publicly vowing now that any newsletter I produce will be designed so that folks can easily view the document on a monitor, as well as preserving the option to print it out.
