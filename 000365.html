I am intrigued by ObjectSpaces

Andy Hopper, who I've talked to many times about code generation, seems to be enamored with ObjectSpaces. Andy claims (and I have no reason to disbelieve him) that it eliminates the need to code-gen object-relational mapping layers. This is very interesting to me, since it removes some of the low-hanging fruit from code generation motivation.

It's time for me to dig into ObjectSpaces. I've been spending most of my Whidbey time on MSBuild, but now it's time to check out something new. More on this soon.

I'm afraid I don't consider Thomas to be an impartial observer, here. He has a commercial product known as EntityBroker, so I expect him to tell you in his own inimitable way that any O/R mapping tool other than his sucks.
Does ObjectSpaces have limitations? Yep. It doesn't create your data objects - you instead need to write them yourself or write a framework that does create them, and it doesn't natively support creating the database schema. It's not as fast as a codegenned O/R layer (they anticipate it to be roughly 40% slower). There are other issues I can see, but this comment's getting too long as it is.
I saw some extremely interesting things at the PDC that make OS more attractive. On Longhorn (yes, unfortunately only Longhorn), you will be able to use the classes in the System.Search namespace to create OPath queries from plain text requests. Imagine a doctor typing "Show me all of the encounters I had with John Lam over the last year."
I see an opportunity for a company to build a tool to fill in the gaps.
