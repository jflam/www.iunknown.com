Comment spam and ineffectiveness of ip address filtering

I've been fighting a battle with the comment spammers for some time now on this blog. Most of you don't notice the comment spam because I get to it fairly quickly. However, one thing that I've realized is that the IP address filtering feature of Movable Type is next to useless, since the same spammer now appears from apparently random IP addresses. I was curious about how this works, and just discovered that it's due to ARP cache poisoning. A fascinating read for any of you who are fighting spam on your own blogs.

Update: From my comments, it does appear that what I'm seeing isn't due to ARP cache poisoning. Doing some quick reverse DNS lookups shows that most of the spam is coming from various ISPs (rr.com and aol.com). So either the spammers have a bunch of "zombie" computers that they 0wn, or they're simply logging in through dial-up connections. Since it looks like the spam is entered by hand, I suspect the latter.

Various fun VB.NET observations

I created some VB.NET code where I needed an array with 1 element. This forced me to have to dimension the array with what looks like zero elements: Dim myArray(0) As MyType.

I'm also having a lot of fun with NUnit these days. One thing that I like to do is write stuff out to the Console. Here's a simple class that lets you write unit tests that assert output to the console:


Public Class ConsoleTester : Inherits Assertion

  Private writer_ As StringWriter
  Private oldWriter_ As TextWriter

  <SetUp()> Public Sub Setup()
    oldWriter_ = System.Console.Out
    writer_ = New StringWriter
    System.Console.SetOut(writer_)
  End Sub

  <TearDown()> Public Sub TearDown()
    System.Console.SetOut(oldWriter_)
  End Sub

  Protected ReadOnly Property ConsoleOutput()
    Get
      Return writer_.GetStringBuilder().ToString
    End Get
  End Property

End Class

Arguments against shared assemblies

When you put an assembly into the GAC, you need to make sure that you never introduce breaking changes into that assembly. However, a particularly serious problem that arises is how the "no breaking changes" rule can severely limit your ability to innovate. Oftentimes when you want to innovate you can only do so by introducing breaking changes (just ask someone on the Windows team about how hard it is to innovate without introducing breaking changes).

So here's another way of looking at whether you should have code in the GAC: is this code going to be stable for a long time? If the velocity of change in that code is going to be high, you are far better off deploying that code as private assemblies.
