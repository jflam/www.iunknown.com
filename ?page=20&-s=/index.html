  • Upgrading my MacBook Pro HDD was fun

    • 13 Jul 2006
    • 4 Responses
    •  views
    • Edit
    • Delete
    • Tags
    • Autopost

    I upgraded my MacBook Pro HDD to a 100GB 7200rpm disk tonight. Before tonight I was using the 80GB 5400rpm disk that shipped with my computer. I was amazed by how straightforward the experience was.

    I began by using Disk Utility to create a copy of my system partition onto an external USB 2.0 HDD. This took a long time – I let it run overnight. Are there better (faster) applications for creating an image of my system partition?

    I then used the most excellent MacBook Pro Disassembly Guide to pull apart my MacBook Pro to physically replace the disk. One thing you shouldn’t do, however, is disconnect the keyboard and trackpad. I just propped the keyboard against the LCD while replacing the HDD.

    I started my Mac and booted the OS from the DVD drive. I created a HFS+ partition on my new disk and installed Mac OS onto it. When I was finished, I plugged in my USB drive, and used the Migration Assistant to copy over my old user settings and files. This took about 1.5 hours and I had a machine that was every bit the same as my original!

    The only app that I had to reinstall was Parallels.

    What a nice experience. And the machine definitely feels much snappier as well.

    Regarding ruby gems, if you were getting this error:

    ERROR: While executing gem … (Gem::GemNotFoundException)

    Jim Weirich posted this solution on ruby-talk earlier this week:

    There was a gem on rubyforge that somehow corrupted the gem index on that site. A lot of people seemed to have downloaded the corrupted index and it has been causing problems. The easiest way to deal with it is to just delete your gem cache file.

    Use the gem env command to find the default gem repository directory and remove the source_cache file there.

    $ gem env gemdir
PATH_TO_DEFAULT_GEM_REPOSITORY
$ rm PATH_TO_DEFAULT_GEM_REPOSITORY/souce_cache
$ rm $HOME/.gem/source_cache

    The .gem cache will only exist if the default repository is not writable from your user account. Delete both if they exist.

    Regarding .NET Framework 3.0, I was getting a really bizarre exception about mismatched library versions when trying to display images using WPF. It turns out the culprit is the Windows Imaging Components. You’ll need to update via this link.

  • JavaScript and Continuation Passing Style

    • 30 Jun 2006
    • 4 Responses
    •  views
    • Edit
    • Delete
    • Tags
    • Autopost

    I’ve been writing a lot of JavaScript code for work recently, and almost all of it involves async operations. Most of the code that I write is using continuation passing style. It’s quite a natural style of programming in JavaScript since functions are first class objects in JavaScript. It’s also interesting that closures and higher order functions work so beautifully together.

    Consider this simple wrapper around XMLHttpRequest:

    function get_url(file, completion) {
var x = window.XMLHttpRequest ? new XMLHttpRequest()
: new ActiveXObject("Microsoft.XMLHTTP");
x.onreadystatechange = function () {
if (x.readyState == 4 && x.status == 200)
completion(x.responseText);
};
x.open('GET', file, true);
if (window.XMLHttpRequest)
x.send(null);
else
x.send();
}

    Most of the cruftiness in this function involves accounting for differences between IE and Firefox, but you can see that the continuation is passed in via the completion parameter, and I’m using a closure to capture the x variable in an anonymous function that’s called when the async HTTP GET request has completed processing. The anonymous function then calls the continuation, passing the result of GET.

    Just last night I was reading this excellent paper on functional programming. It explains what I am doing so much more elegantly than I can (although I hope the code speaks for itself). I highly recommend that you spend some time reading it so that you can get what all of this functional programming hoopla is all about. I assure you, it’s the real deal.
